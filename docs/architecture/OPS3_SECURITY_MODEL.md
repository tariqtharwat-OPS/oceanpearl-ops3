# OPS3 SECURITY MODEL

## 1. Role Boundaries & Strict Separation
Unlike generic CRUD frontends where security relies on soft-UI rendering hidden buttons, OPS3 enforces security directly within the Firebase connection logic mapped exactly to the 10 User Classes of the Blueprint.

**Implementation Strategy:**
- We pass a custom token matching the user identity role enum (e.g., `role_boat`, `role_finance`).
- A `rules_version = '2'` mapped Firestore `.rules` configuration parses the `request.auth.token.role` attribute strictly.
- User objects without valid tokens explicitly error when connecting, routing immediately to `ROLE CONFIG MISSING`.
- No single component permits fallback arrays bridging two role arrays in rendering (Enforced locally via the audited `<roleNavConfig>`).

## 2. Location & Unit Scope Inheritance
Data segregation maps precisely to the Location Hub Builder Tree established in `adm_loc`.

**Physical Matrix Scoping:**
Every person mapped to `adm_ppl` belongs either strictly to a Hub (e.g., Kaimana) or explicitly to a unit instance (e.g., Boat Faris).
- A Boat Operator writes to `/locations/{hub_id}/units/{boat_id}/trips/{trip_id}/transactions/`. They have `read, write` ONLY restricted explicitly to their parent `unit_id`.
- A Factory context works exactly identically bound to its distinct unit ID scope.
- A Location Manager (Kaimana) inherits `read, write` bounds universally to `/locations/kaimana/units/{any}` via a top-level pathing override block.
- An Office Admin (`role_office`) or Financial Board Officer (`role_finance`) possesses `read` global attributes aggregating the `.collectionGroup('transactions')`, but ONLY inherits explicitly defined global `write` methods specifically to AP/AR collections.

## 3. The Two-Phase Separation Mechanism (Transfers)
You cannot modify another unit's data scope explicitly. 

**Workflow Security Flow (DO Transfers):**
1. Factory A writes `fac_trf` (Document TRF -> Target Cold Storage).
2. The `POSTED` dispatch generates a mathematically locked shadow placeholder directly inside the Cold Storage subcollection via an elevated backend Cloud Function trigger, bypassing direct write scope.
3. Cold Storage explicitly accepts the shadow placeholder under their valid UI view (`cs_recv`).
This completely prevents cross-scope payload manipulation from either interface.

## 4. Shark AI Inheritance Engine
The OPS3 internal LLM companion engine (`role_shark`) acts primarily as an *observer*.
To prevent contextual drift or inappropriate exposure, the Shark's context engine explicitly inherits the requestor's token map.
If a Factory Operator asks Shark a query about Boat B1's expected catch margins, Shark explicitly yields an Access Denied error based on the bounds constraint generated by the Factory token context. It observes exactly what the caller is legally allowed to see. Only the System Admin, Finance, or CEO invoke the complete graph schema query parameters.
