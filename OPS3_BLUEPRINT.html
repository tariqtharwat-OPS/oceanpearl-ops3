1. System Core Blueprint

The immutable foundation of OPS3. Goods and Money Control.

A. Operational Hierarchy

COMPANY (Ocean Pearl)
├── LOCATION (Kaimana)
│ ├── UNIT (Boat Faris) - Captures Raw Stock, holds Trip Wallet
│ ├── UNIT (Factory) - Transforms Raw -> Finished, holds Factory Wallet
│ └── UNIT (Office) - Absorbs localized OPEX, holds Petty Cash
└── LOCATION (Surabaya)
└── UNIT (Cold Storage) - Holds finished stock prior to export sales

Rule: Units belong to Locations. Inventory belongs exclusively to Units. Operations occur at the Unit level. Approvals occur at the Location level.

B. Role Scopes

AdminFull system override, settings, database tools, user management.
CEOOmni-directional read access. Global dashboards, cross-location transfers, anomaly resolution.
Location ManagerSees only their assigned Location's aggregation. Approves local unit expenses and local transit.
Unit OperatorStrict action-floor context. Cannot see financials of other units. Can only Receive, Process, Transfer, Waste.
Finance OfficerGlobal ledger view. Manages Payables, Receivables, inter-wallet liquidity, and bank reconciliations.
InvestorRead-only, high-level sanitized dashboards. Revenue, margins, inventory valuation.

C. Wallet & Ledger Architecture

  • No Silent Balances: Every mutation must be backed by a double-entry ledger record.
  • No Negative Stock: Transactions mathematically causing negative unit inventory will throw a hard DB block.
  • Wallet Flows: Bank -> Company Wallet -> Location Wallet -> Unit Wallet -> Expenses/Suppliers
  • Document Lifecycle: DRAFT (No Ledger Posting) -> POSTED (Locked, Posts to Ledger) -> CANCELLED (Creates Reversal Posting)

5. Shark AI Architecture

Operational Intelligence

A. Context & Inheritance Model

Shark is not a standard RAG chatbot. Shark executes within an exact replica of the logged-in User's IAM scope.

  • Operator asks: "What's my stock?" -> Shark retrieves ONLY Unit stock.
  • CEO asks: "What's my stock?" -> Shark aggregates entire company.

B. Proactive Risk Horizons

Shark runs background CRON jobs evaluating Ledger and Inventory invariant rules:

Risk 1: Yield Anomaly
> Detects if MAW-DRY yield drops below 1.5% from GROUPER-RAW processing batch.
Risk 2: Wallet Irregularity
> Detects large expense disbursements (>Rp50M) not tied to a supplier PO.
Risk 3: Price Spikes
> Detects if receiving input price is +15% above 7-day trailing average for the same Location.

6. Execution Roadmap

Implementation phases following strict blueprint approval.

Phase 1: Foundation & Roles

Establish the `RoleBasedRouter`. Generate structural boundaries for Admin, CEO, Operations, Finance, and Investor so layouts cannot mix. Setup Tailwind styling and React component hierarchies.

Phase 2: The Industrial Forms & Frontend Flow

Building the multi-line React forms for Receiving, Processing, Transfer, and Expenses. Ensuring local state calculates Grand Totals correctly. Building the physical A4 print layouts.

Phase 3: Ledger Integration & Backend Execution

Rebuilding `workflows.js` on the Cloud Functions to accept the structured multi-line payloads. Enforcing the double-entry accounting constraints (debts equal credits) and blocking negative inventory mutations.

Phase 4: Executive Pulse & Shark AI

Developing the CEO and Finance read-only dashboards aggregating the new ledger data. Injecting the Shark AI interface and establishing CRON jobs for background anomaly monitoring.

Anti-Drift Commitment

Coding will only proceed strictly following this approved blueprint. No undocumented logic will be deployed. Any new screen requests must be added to this document first.
